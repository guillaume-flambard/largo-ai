# Building an AI usage policy

An AI usage policy is not a compliance document written by legal and filed in a drawer. It is an operational tool that developers consult to know what they can do, what they cannot do, and what requires sign-off. A useful policy fits on 2 pages. A useless one runs to 20 that nobody reads.

## The five clauses of an operational policy

**1. Scope of approved tools**

List the tools validated by the organisation (with their permitted usage level) and those that are out of scope. Example:

| Tool | Status | Conditions |
|---|---|---|
| Claude Code (Anthropic) | Approved | On internal and non-confidential code |
| Cursor | Approved | Same |
| ChatGPT (OpenAI) | Approved with restrictions | See data clause |
| Unlisted tools | Subject to approval | Ask [AI lead] before use |

**2. Data prohibited in prompts**

This is the most critical clause. List explicitly what cannot be sent to an external model:

- Personal data of customers or users (names, emails, bank details, health data)
- Non-public company secrets (roadmap, unpublished financial data)
- Technical secrets (API keys, tokens, passwords, certificates)
- Code that exposes the logic of critical proprietary algorithms

And what is permitted: internal code without sensitive data, feature specs, generic technical questions, marketing content.

<Idee>
The simple rule for developers: "Before sending something to an external model, ask yourself whether you would be comfortable if that text were public." If not, don't send it. This heuristic covers 90% of cases without having to memorise the exhaustive list.
</Idee>

**3. Responsibility for outputs**

Code, text, or decisions produced with the help of an agent remain the responsibility of the developer. This means:

- The developer reads and validates the generated code before committing it
- A "generated by AI" comment in the code does not relieve the author of responsibility
- Agent outputs are not proof, certifications, or legal validations

**4. Declaration of high-risk uses**

Any project that plans to use AI for decisions about people (recruitment, evaluation, scoring) must be declared to the AI lead before development begins. The AI lead assesses the AI Act risk level and initiates the compliance process if necessary.

**5. Annual review**

The policy is revised every 12 months. The AI lead is responsible for updates. Developers are notified of significant changes.

<Exemple>
**AI usage policy — short version (template)**

*Effective date: [date]. Next review: [date + 12 months]. Lead: [name/email].*

**Permitted:** using Claude Code, Cursor, Copilot on internal code that does not contain the data listed below. Generating tests, specs, documentation. Using AI assistants for generic technical questions.

**Prohibited:** sending personal customer data, infrastructure secrets, unpublished financial data to an external model. Using an unlisted tool without prior approval.

**Subject to approval ([AI lead]):** any project developing an AI system that assists decisions about people. Any external AI tool not yet validated.

**Responsibility:** all AI output is the responsibility of the developer who integrates it. Review and validation remain mandatory.
</Exemple>

## What the policy alone cannot solve

A policy written without adoption is paper. For it to be operational:

- It must be co-built with the teams (developers know which uses actually exist)
- The AI lead must be reachable and respond within 24 hours on "subject to approval" cases
- Non-compliance incidents must generate shared learning, not systematic sanctions for first-time cases

<Attention>
Do not write a policy that prohibits what is already being practised without offering an alternative. A policy that says "prohibited" without providing a legitimate framework generates Shadow AI — developers keep going, but underground. The goal is to get out of the shadow, not to push people further into it.
</Attention>

<Exercice>
Draft the "prohibited data" clause of your AI usage policy, listing the categories specific to your context (which customer data, which technical secrets, which intellectual property). Test the clause by asking three developers to judge 5 concrete cases: "do I send this?" Adjust if answers diverge — ambiguity in the clause is an operational risk.
</Exercice>
