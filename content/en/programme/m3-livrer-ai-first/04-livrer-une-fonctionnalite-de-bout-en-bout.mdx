# Shipping a feature end to end

Here's a complete walkthrough. Not a theoretical example — the real sequence of an AI-first delivery, with the decisions at each step.

## The scenario

**Request**: Add an "email notification" feature when a user is mentioned in a comment.

## Step 1 — Feature request → Breakdown

Before any spec, we decompose:

1. Detect mentions in comment text (`@username`)
2. Resolve usernames to email addresses
3. Send the email (sending service)
4. Store notifications to prevent duplicates
5. API Route to trigger everything
6. Unit tests on detection and deduplication

Six atomic tasks. We start with the first.

## Step 2 — Spec → Agent → Diff

**Spec 1**:

> **Context**: `src/utils/` contains pure utilities. No external dependencies in this folder.
> **Objective**: Create `src/utils/mentions.ts` with an `extractMentions(text: string): string[]` function that returns the list of mentioned usernames (format `@username`, letters/digits/underscores, length 2-20).
> **Constraints**: Pure function, no imports. Return unique values in lowercase.
> **Success criterion**: Unit tests pass on the cases listed in the spec.

The agent produces the function and a test file. We read both diffs.

The test covers the nominal cases but not mentions at the start of a string and false positives like `email@domain.com`. We update the spec and relaunch. The second diff is correct.

Commit.

<Idee>
The MR is the unit of truth. Not the prompt you wrote, not the intention you had — it's the diff you read, understood, and decided to merge. If you can't defend every line of the diff, you're not ready to open the MR.
</Idee>

## Step 3 — Handling an agent failure

On Spec 4 (sending service), the agent proposes installing `nodemailer` while your stack already uses Resend. The diff introduces an unjustified dependency.

Wrong reaction: "no, use Resend, here's how…" via re-prompting.

Right reaction: update `CLAUDE.md` (add "email service: Resend via `src/services/emailService.ts`, do not install a new email package") and rewrite the constraint in the spec.

<Exemple>
**Re-spec, not re-prompt**:

> **Constraints (updated)**: Use exclusively `src/services/emailService.ts` (existing Resend wrapper). Do not add any dependency. The signature is `emailService.send({ to, subject, html })`.

The agent relaunched on this spec produces the correct code on the first try.
</Exemple>

## Step 4 — Generating tests

Once all six specs are committed, we give the agent a test spec:

> **Context**: The functions `extractMentions`, `resolveMentions`, `deduplicateNotifications` are implemented.
> **Objective**: Generate an integration test file covering the full flow: text with mentions → emails sent → no duplicate if the same comment is reprocessed.
> **Constraints**: Use Vitest. Mock `emailService.send`. Do not call a real database.
> **Success criterion**: The full-flow test is green. The `emailService.send` mock is called exactly N times for N unique mentions.

We read the test file diff. We verify that the assertions test behavior, not implementation.

## Step 5 — MR

The MR contains:
- The six atomic commits (one per spec)
- The integration test file
- The `CLAUDE.md` update

The MR description recaps the breakdown and references the specs. Any team member can understand the architectural decision by reading the MR without having followed the session thread.

<Attention>
Don't group six specs into a single "feat: notifications" commit. One commit per spec lets you bisect cleanly if something breaks, and revert a single decision without undoing the rest.
</Attention>

<Exercice>
Pick a small feature (2-3 days of usual work). Deliver it entirely AI-first: breakdown → atomic specs → diffs read and committed → MR with description. Measure the time spent writing specs vs the time you'd normally spend coding. Note what the agent missed and what was missing from the context.
</Exercice>
