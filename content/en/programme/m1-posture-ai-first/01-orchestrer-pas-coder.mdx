If you've ever pasted code generated by ChatGPT into a file without reading it
in full, you've practiced Shadow AI. That's not a criticism — it's the norm in
most teams today. The problem is that it looks like productivity while accumulating
debt and vulnerabilities that nobody sees coming.

AI-first is the antithesis of Shadow AI: it's a **method** for getting the most
out of code agents without sacrificing quality or security.

## The core idea

<Idee>
**Orchestrating ≠ coding.** In AI-first mode, the developer doesn't disappear: they
shift role. From "the person who writes the code" to "the person who frames the task,
supervises the agent, and validates what goes to prod." Same level of technical
rigor, applied differently.
</Idee>

The distinction matters because it changes what you need to be good at. Writing code
is one skill among many. Framing a task for an agent, reading a diff with the eye of
a code reviewer, and deciding "this ships" or "this doesn't ship" — these are skills
good developers already have. They just apply differently now.

## What changes in a workday

Concretely, working AI-first looks like this:

1. You write a short spec (context, objective, constraints, success criteria) — two to
   ten lines.
2. You hand the spec to an agent (Claude Code, Cursor), making sure it has the right
   project context.
3. You read the produced diff: every line, every modified file.
4. You iterate or approve, then commit.

What changes: step 3 becomes the main work, not step 2. And that's where your expertise
matters most — no agent knows better than you what's acceptable in your codebase.

## The invisible debt of Shadow AI

<Attention>
Uncontrolled AI usage generates technical debt faster than writing by hand, because
the produced code is plausible but rarely optimized for your context. An agent that
doesn't know your conventions will introduce a third auth pattern into a project that
already has two. Without systematic review, you won't see it until it becomes a problem.
</Attention>

This isn't a tooling problem — it's a methodology problem. The modules that follow cover
exactly that: how to give an agent the right context, how to structure specs, how to read
and validate results.

## An example of the difference

<Exemple>
**Shadow AI:** "Generate a function that paginates my user list."
The agent produces something. You copy-paste it. It works locally. You push. The week
after, you discover it used an unindexed SQL query pattern that holds at 100 users but
collapses at 10,000.

**AI-first:** You write: "Add pagination to `UserRepository.findAll()`.
Constraints: use the cursor-based pattern already present in `OrderRepository`,
no SQL offset, max page size 50. Return a `PaginatedResult<User>` object.
Tests to add in `user.repository.spec.ts`." The agent has context. The diff is
readable. You verify the pattern matches what you specified before merging.
</Exemple>

## Your turn

<Exercice>
Take the last dev task you did with AI. Answer these three questions:

1. Did you specify the technical constraints (existing patterns, expected performance,
   project conventions)?
2. Did you read every line of the diff before committing?
3. Is there a piece of code in that diff you wouldn't have written yourself and whose
   implications you didn't check?

The goal isn't to feel guilty — it's to calibrate the gap between where you are now
and a structured AI-first workflow.
</Exercice>

## Summary

- AI-first = orchestrating agents, not just querying them. The developer stays the
  decision-maker on every diff.
- Shadow AI = usage without method or guardrails. Plausible short-term, risky medium-term.
- The mindset shift doesn't demand less technical rigor — it redirects it toward framing
  and review.
