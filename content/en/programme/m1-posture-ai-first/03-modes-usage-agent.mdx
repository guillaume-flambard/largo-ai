Not all AI usage is equal in terms of risk and control. Accepting an inline completion
suggestion in a function you just wrote is very different from launching an agent in
autonomous mode over an entire module. Conflating the two means either underusing the
tool (too cautious) or producing unmanageable diffs (too permissive).

## The three modes

<Idee>
**Mode 1 — Inline completion:** The AI suggests code as you type (Cursor, GitHub
Copilot). You accept or reject each suggestion. This is the lowest level of autonomy
— context stays in your head, you stay in control at every keystroke.

**Mode 2 — Supervised autonomous agent:** You describe a task. The agent reads files,
plans steps, modifies code, runs commands. You supervise the steps and review diffs
before approving. Claude Code operates primarily in this mode.

**Mode 3 — Orchestration:** A high-level agent decomposes a complex project and
delegates to specialized sub-agents. This is an advanced pattern to know about — not
the starting mode.
</Idee>

## Choosing the right mode for the task

| Task type | Recommended mode | Reason |
|---|---|---|
| Complete a function you're writing | Completion | Trivial context, low risk |
| Add an API endpoint on a known project | Supervised agent | Multiple files, defined scope |
| Write tests for an existing module | Supervised agent | The diff is readable and testable |
| Cross-cutting architecture refactor | Supervised agent + enhanced review | High breakage risk |
| Full pipeline feature → CI → deploy | Orchestration (advanced) | Complex multi-step pattern |

## Mandatory checkpoints

Regardless of mode, two things are never delegated:

1. **Reviewing the diff.** Every modified file, every line. An agent can produce
   something functional that breaks a domain invariant — only someone who knows the
   project can catch that.
2. **The commit decision.** You sign the commit, not the agent. You are responsible
   for what goes to prod.

<Attention>
In autonomous agent mode, watch out for the "correction spiral": the agent makes a
mistake, you ask it to fix it, it fixes it but introduces another problem, and so on.
If you're on the third correction round for the same task, stop. Take back control
manually or rewrite the spec from scratch with more explicit constraints. The
correction spiral is almost always a sign of a spec that's too vague, not an
incompetent agent.
</Attention>

## On multi-agent orchestration

Orchestration (one agent driving other agents on parallel sub-tasks) is a real and
powerful pattern. It's also complex to configure correctly: errors propagate between
agents, debugging is harder, and human control is more indirect.

<Exemple>
A typical orchestration workflow (for illustration, not a prerequisite):

- Agent 1 (planner): decomposes the spec into independent sub-tasks.
- Agent 2 (backend): implements the API endpoints.
- Agent 3 (tests): writes tests against what Agent 2 produced.
- Agent 1 (validator): checks consistency and flags conflicts.

The value is real on large projects. But mastering modes 1 and 2 is a prerequisite:
you don't start with orchestration.
</Exemple>

## Your turn

<Exercice>
On your next dev ticket, consciously choose the mode:

- If the task touches 1-2 functions in a file you're already editing →
  inline completion.
- If the task touches multiple files with a clearly defined scope → supervised
  agent, spec written before launching.
- If you're tempted to launch the agent without writing the spec first → go back to
  the lesson on specs (module 2) first.

Note the chosen mode and why. After the task, evaluate: was the result better or worse
than expected? Why?
</Exercice>

## Summary

- Completion, supervised agent, and orchestration are three distinct levels of autonomy
  with different risks and guardrails.
- Supervised agent mode is the heart of daily AI-first practice.
- Orchestration is a pattern to know about, not an immediate objective.
- In all modes: diff review and commit decision remain human.
