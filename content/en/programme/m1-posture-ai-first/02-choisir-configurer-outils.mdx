The tool choice is rarely the problem. What makes the difference is configuration:
a poorly configured agent in a project with no context will produce generic code
that ignores your conventions. A well-configured agent in a project that gives it
the right signals can ship something mergeable on the first try.

## The two main tools

<Idee>
**Claude Code** is a CLI agent: it runs in your terminal, reads and modifies your
files, runs commands, and can operate in supervised autonomous mode. It's ideal for
tasks that touch multiple files, cross-cutting refactors, and end-to-end feature additions.

**Cursor** is an AI-first editor: it embeds a model directly in the IDE, with
contextual completion, chat anchored to selected files, and a "Composer" mode that
can apply multi-file changes. It's ideal for fine-grained editing, questions about
existing code, and small in-context modifications.
</Idee>

The two aren't competing: they complement each other. Claude Code for autonomous agent
tasks, Cursor for daily editing. Many AI-first developers use both on the same day,
depending on the nature of the task.

## What the tool needs to know about your project

An agent without project context is like a junior developer arriving on day one with
no onboarding. They'll produce something — but not necessarily in the right framework
version, not in the right pattern, and maybe with dependencies that conflict with yours.

The minimal context an agent needs:

- **Which framework / which version** (Next.js 15, not Next.js 13 — the APIs differ)
- **Project conventions** (where types live, how files are named, import style)
- **Existing patterns** (how you handle auth, how you structure components, which ORM
  and how you use it)
- **What not to do** (anti-patterns you've banned, libraries not to add)

All of this goes in a `CLAUDE.md` file at the project root for Claude Code, or in
the Cursor project rules. The next module is entirely dedicated to these context files.

## Minimal Claude Code configuration

<Exemple>
After installation (`npm install -g @anthropic-ai/claude-code`), the first thing to
do in a new project:

1. `claude` — launch the interactive agent
2. `> /init` — Claude Code analyzes your project and generates a starting `CLAUDE.md`
3. Re-read the file and complete it with your actual conventions (it won't know all of
   them — it only works from the visible code)

The `CLAUDE.md` generated by `/init` is a starting point, not a deliverable. You'll
enrich it across sessions whenever you notice an agent has missed a convention.
</Exemple>

## Minimal Cursor configuration

In Cursor, project context is configured in `.cursor/rules` (`.mdc` or `.md` files
depending on the version). The logic is identical to `CLAUDE.md`: you describe the
project, conventions, and what's forbidden.

One important point: in Cursor, you can anchor a conversation to specific files
(`@filename.ts`). This is useful for asking precise questions about a module without
the agent hallucinating about the rest of the project.

## Choosing the right LLM: the engine behind the tool

The tool (Cursor, Claude Code, Aider, GitHub Copilot) is just the body. The engine is the large language model (LLM) you choose to power it. Today, you are no longer limited to a single model. Choosing the right LLM depends on three factors: task complexity, cost (or speed), and the size of the required context.

We can divide coding models into three major families:

### 1. Reasoning / Thinking Models
*Examples: OpenAI o1 / o3-mini, DeepSeek-R1.*
* **How they work:** Unlike classical LLMs that generate their response word-by-word immediately, reasoning models go through an invisible "thinking" phase (chain of thought). They plan, test hypotheses, and correct their own mistakes before writing any code.
* **When to choose them:** Designing a complex architecture, writing a difficult optimization algorithm, or debugging a particularly tricky multi-file bug.
* **Why:** They drastically reduce error rates on complex tasks, at the expense of higher latency and cost.

### 2. High-End / Frontier Models
*Examples: Anthropic Claude 3.5 / 3.7 Sonnet, OpenAI GPT-4o, Google Gemini 1.5 Pro.*
* **How they work:** Extremely versatile models, fast to respond, and trained on massive amounts of high-quality code.
* **When to choose them:** For daily development in agent mode, generating standard features (CRUD, forms, API routes), and writing tests.
* **Why:** **Claude 3.5/3.7 Sonnet** is the current benchmark for syntactic precision, strict instruction following, and navigating a codebase. **Gemini 1.5/2.5 Pro** stands out when you need to pass a massive amount of technical documentation or files, thanks to its context window of over a million tokens.

### 3. Fast and Lightweight Models
*Examples: Google Gemini 2.0 Flash, Anthropic Claude 3.5 Haiku, OpenAI GPT-4o-mini.*
* **How they work:** Smaller, ultra-fast, and very inexpensive models.
* **When to choose them:** For real-time inline code completion (auto-complete), automated code reviews on CI, or quick syntax questions.
* **Why:** Their near-zero latency allows you to code without breaking your workflow.

---

### LLM Quick Decision Table

| Model / Family | Ideal Use Case | Strengths | Weaknesses |
|---|---|---|---|
| **Claude Sonnet** (3.5/3.7) | Daily Development (Agent / Composer) | Code precision, convention adherence | Moderate cost |
| **Reasoning** (o3-mini, R1) | Complex algorithms, complex bugs, architecture | Pure logic, self-correction | Slow, no inline completion |
| **Gemini Pro** (1.5/2.5) | Reading giant codebases or massive docs | Giant context (1M+ tokens), multimodal | Sometimes less rigorous on fine syntax |
| **Flash Models** (Gemini Flash, Haiku) | Inline completion, simple scripting, CI | Speed (ultra-low latency), very low cost | Limited reasoning capacity |

## Practical comparison for choosing

| Task | Claude Code | Cursor |
|---|---|---|
| Add a feature (multiple files) | ✓ agent mode | ✓ Composer |
| Cross-cutting refactor (rename, pattern) | ✓ autonomous agent | Possible but slow |
| Question about an existing file | Possible | ✓ anchored chat |
| Fine editing / completion | No | ✓ inline |
| CI pipeline / scripts | ✓ | No |

<Attention>
Don't confuse "completion mode" and "agent mode." In completion, the AI suggests
code line by line — you stay in control at every keystroke. In agent mode, the AI
executes an autonomous sequence of actions on your files. These are very different
levels of autonomy that require different guardrails. The next lesson covers this
in detail.
</Attention>

## Summary

- **The tool is just the body, the LLM is the engine.** Select the model based on complexity and need (reasoning for complex tasks, generalist for daily dev, fast for inline completion).
- Claude Code = autonomous CLI agent, strong on multi-file tasks and end-to-end workflows.
- Cursor = AI-first editor, strong on contextual editing and questions about code.
- Both require a project context file to produce relevant output in your codebase.
- Configuration isn't a one-shot: you enrich it across sessions.
