# AI-assisted code review

Code generated by an agent is code. It must be reviewed as code. The most common mistake in teams adopting the AI-first workflow is relaxing review rigor because "the agent doesn't make dumb mistakes." The agent makes different mistakes — often more subtle ones.

## Why AI code needs more review

A human developer implicitly carries the project's constraints: they know this table is sensitive, this API is public, this pattern was abandoned after an incident. The agent only knows what's in its context. What isn't in the context can become a vulnerability in the diff.

Typical problems in AI-generated code:

- A route created without an authentication middleware (the agent forgot it because the example in its spec didn't have one)
- A dependency added without justification (the agent picked the most well-known library, not the one in your stack)
- An inconsistent error-handling pattern (the agent followed an example it found in another file, not yours)
- Logs that expose sensitive data (the agent logged the entire object to ease debugging)

<Idee>
The human developer is the sole final decision-maker on what enters the codebase. The agent assists the review — it can explain a section, identify edge cases, suggest tests — but the decision to accept or reject belongs to the developer.
</Idee>

## The review checklist for AI-generated code

<Exemple>
**PR Checklist — AI-generated code**

**Correctness**
- [ ] Logic matches the spec (not what the agent interpreted)
- [ ] Edge cases are handled (null, empty, out-of-range)
- [ ] Types are correct and consistent with existing types

**Security**
- [ ] No unplanned unauthenticated route or endpoint
- [ ] No sensitive data in logs
- [ ] No SQL query built by concatenation
- [ ] User inputs are validated before use

**Architecture**
- [ ] Code respects the existing layer separation
- [ ] No unjustified new dependency in package.json
- [ ] Error-handling pattern is consistent with the rest of the project

**Coverage**
- [ ] Tests cover behavior, not implementation
- [ ] Error cases are tested, not just the happy path
</Exemple>

## Using the agent to assist review

The agent can help you review its own diff — or another agent's. Recommended usage:

- "Walk me through what this function does line by line"
- "What edge cases does this implementation not handle?"
- "Are there any injection vectors in this code?"

What the agent cannot do for you:

- Decide whether the architectural pattern is acceptable in your context
- Evaluate whether an added dependency complies with your policy
- Validate that the business logic is correct according to your business rules

<Attention>
Never ask an agent "is this diff good?" and treat the result as validation. The agent will probably say "yes" while listing the best practices that were followed. It doesn't know your undocumented constraints, your past incidents, your unwritten architectural decisions. "LGTM from the AI" is not a review.
</Attention>
