# Security, secrets, and sensitive data

The AI-first workflow introduces an attack surface traditional development doesn't have: the agent's context. Everything you put in a spec, a `CLAUDE.md`, or a message to the agent is treated as text and can be logged, memorized, or transmitted. The rule is simple and has no exceptions.

## The cardinal rule

**Nothing secret enters an agent's context.**

No API key. No password. No token. No `.env` file contents. No database connection string. No real users' personal data.

The context is not your terminal. It's text sent to an external service.

<Idee>
An agent's context is a security boundary. What enters the context can exit it — in logs, in future training data, in a poorly worded response. Treat the context like an unencrypted channel: only put in it what could be public.
</Idee>

## The most common leak vectors

**1. The .env file pasted into a spec**

Scenario: you write a spec for a Stripe integration and paste your `.env` file "so the agent understands the available configuration." The Stripe key is now in the context.

Correct alternative: describe environment variables by name only.

> "The application uses `STRIPE_SECRET_KEY` (string, available in the environment). Do not hardcode any value."

**2. Production data in examples**

Scenario: to illustrate the expected data format, you paste a row from your production `users` table. It contains a real email, a real name.

Correct alternative: manufacture explicitly fictional data.

> "Format: `{ id: 'usr_abc123', email: 'alice@example.com', name: 'Alice Example' }` (fictional data)"

**3. Database schema with sensitive migrations**

Drizzle or Prisma schemas: the schema itself (types, columns, relations) is generally safe. What isn't: migration files containing real seed data, or comments that reveal sensitive architectural decisions ("this column hides the fact that…").

**4. Git history as a vector**

If a secret was committed — even in a commit you subsequently reverted — it's in the git history. If you give an agent access to your repo (some workflows do this), that secret is accessible. Revoke any committed secret immediately, even if it was only there briefly.

<Exemple>
**Concrete rule to encode in CLAUDE.md**:

```markdown
## Security
- Never generate secrets, tokens, or keys hardcoded in the code
- Environment variables are referenced by name only (e.g.: process.env.STRIPE_SECRET_KEY)
- If you need a configuration value to generate code, ask me for the variable NAME, not its value
- No real user data in examples or test fixtures
```
</Exemple>

## When the agent produces code with a hardcoded secret

It happens — the agent "completes" an example with a fictional value that looks like a real key.

Procedure:

1. Do not commit the file
2. Check whether the value is a real key (look up the format in the service's documentation)
3. If real: revoke immediately via the service's console, even if the file never left your machine
4. Correct the spec to explicitly forbid this behavior
5. Relaunch the agent

<Attention>
Pasting a database dump, even partial, into an agent context to "show the data structure" is a serious security fault if the table contains personal data. Always use fictional, anonymized data to illustrate formats. Production data stays in production.
</Attention>

<Exercice>
Audit your recent specs and your current `CLAUDE.md`. Look for: variable names that resemble real keys, data examples that could be real, references to configuration files with their values. Note what you find. Write a "Security" section for your `CLAUDE.md` that explicitly forbids these practices.
</Exercice>
