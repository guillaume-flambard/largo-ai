# MCP: connecting agents to tools

The Model Context Protocol (MCP) is the layer that lets an agent go beyond text: it can read files, call APIs, interact with a browser, query a database — anything you expose via an MCP server. It is the step that turns an "assistant" agent into an "actor" agent.

## What MCP is (and what it isn't)

MCP is an open protocol (Anthropic, 2024) that standardises how a model accesses external tools. Instead of embedding tools in each host application (Claude.ai, Claude Code, Cursor…), tools are exposed by an MCP server that any compatible client can consume.

A useful analogy: MCP is to AI what LSP is to code editors — a standard interface that decouples the server (the tool) from the client (the model).

<Idee>
An MCP server can expose three types of primitives: **tools** (actions the agent can execute), **resources** (data the agent can read), and **prompts** (reusable instructions). In engineering practice, tools are the central use case.
</Idee>

## Configuring a local MCP server (Playwright example)

Claude Code reads its MCP configuration from `.claude/settings.json` (or `~/.claude/settings.json` for global config):

```json
{
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": ["@playwright/mcp@latest"]
    }
  }
}
```

After restarting Claude Code, the agent has access to Playwright tools: `browser_navigate`, `browser_click`, `browser_fill`, `browser_screenshot`, `browser_snapshot`…

```bash
# Verify the server is loaded
claude mcp list
# playwright: connected (5 tools)
```

## Driving browser QA via MCP Playwright

Concrete scenario: your agent just generated a login form. You want to validate the real behaviour (rendering, error states, basic accessibility) without writing manual e2e tests.

Instruction to the agent:

```
Via MCP Playwright, navigate to http://localhost:3000/login.
1. Take an initial screenshot.
2. Submit the empty form — verify that validation error messages appear.
3. Fill in invalid credentials — verify the error message.
4. Fill in the test credentials (user: test@example.com / pass: Test1234!) — verify the redirect to /dashboard.
Report: list of observed behaviours + anomalies.
```

<Exemple>
**Typical report produced by the agent after the Playwright session:**

```
Browser QA — /login

✓ Initial screenshot: form rendered correctly, labels visible
✓ Empty submission: "This field is required" shown on email and password
⚠ Invalid credentials: generic "Error" message without detail — recommended: "Email or password incorrect"
✓ Valid login: redirect to /dashboard (302 → 200)
⚠ Password field: no autocomplete="current-password" attribute — accessibility/password manager impact

2 minor anomalies detected.
```

The agent did in 90 seconds what a manual e2e test would have taken 20 minutes to write and stabilise.
</Exemple>

## Beyond the browser: connecting your knowledge bases and databases

Playwright is just the beginning. MCP standardises the connection to any resource. In a production engineering environment, you will want to connect your databases, external API documentations, and internal notes.

Here is how to configure three essential MCP servers in your global settings file (e.g., `.claude/settings.json` or Cursor's MCP configuration):

### 1. Obsidian MCP (Knowledge Access)
To allow the agent to read your internal architecture documentations, specs, and notes stored in your Obsidian vault without saturating the initial context:
```json
"obsidian": {
  "command": "npx",
  "args": ["-y", "obsidian-mcp-server", "--vault-path", "/Absolute/Path/To/Your/Obsidian-Vault"]
}
```
*The agent will then have tools like `search_notes`, `read_note`, or `get_backlinks` to navigate your domain notes.*

### 2. PostgreSQL MCP (Database Schema Access)
To let the agent understand the exact structure of your database without having to paste obsolete schemas by hand:
```json
"postgres": {
  "command": "npx",
  "args": ["-y", "@modelcontextprotocol/server-postgres", "--connection-string", "postgresql://user:pass@localhost:5432/mydb"]
}
```
*The agent can call `describe_table` or `list_tables` to inspect the database in real-time and generate error-free SQL or ORM queries.*

### 3. Fetch/API MCP (External Documentation Access)
To allow the agent to browse the web or parse Swagger/OpenAPI documentation directly without copy-pasting:
```json
"fetch": {
  "command": "npx",
  "args": ["-y", "@modelcontextprotocol/server-fetch"]
}
```

### Complete Multi-Server Configuration Example

```json
{
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": ["-y", "@playwright/mcp@latest"]
    },
    "obsidian": {
      "command": "npx",
      "args": ["-y", "obsidian-mcp-server", "--vault-path", "/Users/dev/obsidian-vault"]
    },
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres", "--connection-string", "postgresql://localhost:5432/db"]
    }
  }
}
```

## Scope and limits of MCP permissions

MCP with Playwright locally is safe: the agent controls a browser on your machine. In shared or CI environments, the rules change:

- Limit accessible origins (do not allow navigation to uncontrolled external domains)
- Disable file-write tools if the server runs in an unsupervised context
- Audit the exposed tools: each MCP tool is an action the agent can chain without asking you

<Attention>
An agent with MCP can call tools in cascade in unanticipated ways. If you expose a "write to database" tool and a "send email" tool, the agent may decide to send the email after writing — without you having explicitly asked for it. Expose the minimum necessary for the current task, not all your available tools.
</Attention>
